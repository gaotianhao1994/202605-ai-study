# AI 开发环境规则

## 核心原则

**优先使用虚拟环境**：在进行 Python 开发时，必须优先激活和使用项目虚拟环境，避免污染全局 Python 环境。

## 规则说明

### 1. 虚拟环境使用规则

- **安装依赖**：必须先激活虚拟环境，再使用 `pip install` 安装依赖
- **运行脚本**：必须先激活虚拟环境，再运行 Python 脚本
- **验证环境**：执行命令前应确认当前所在环境

### 2. 激活虚拟环境

在项目目录下执行：

```bash
# 激活虚拟环境
source venv/bin/activate

# 或使用项目自定义脚本（如果有）
```

### 3. 命令执行检查清单

执行以下命令前，必须确保已激活虚拟环境：

- [ ] `pip install <package>`
- [ ] `python <script>.py`
- [ ] `python -m <module>`
- [ ] `pytest` 或其他测试命令

### 4. 虚拟环境检查方法

执行命令前，可使用以下方式检查当前环境：

```bash
# 查看当前 Python 路径
which python

# 或查看 pip 路径
which pip
```

如果路径包含 `venv` 或项目目录名，则表示在虚拟环境中。

### 5. 依赖管理

- 使用 `requirements.txt` 管理项目依赖
- 安装依赖时记录到 requirements.txt：
  ```bash
  pip install <package>
  pip freeze > requirements.txt
  ```

### 6. 常见错误处理

**错误**：`ModuleNotFoundError: No module named 'xxx'`
**原因**：可能未激活虚拟环境，或在全局环境中安装了依赖
**解决**：确认当前环境，并重新在虚拟环境中安装依赖

---

## 为什么使用虚拟环境？

### 是什么？
虚拟环境是一个独立的 Python 运行环境，每个项目可以有自己的依赖包和版本，互不影响。

### 为什么？
1. **避免依赖冲突**：不同项目可能需要同一包的不同版本
2. **保持全局环境干净**：不在全局环境安装临时或项目专用的包
3. **便于部署和复现**：明确项目依赖，保证环境一致性

### 追问：为什么不直接用全局环境？
- 全局环境只有一个 → 多个项目需要同一包的不同版本时会冲突
- 全局环境的包可能被系统或其他工具依赖 → 随意修改可能导致系统不稳定
- 虚拟环境可以随时删除重建 → 试错成本低
